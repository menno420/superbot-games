## TOP 5 open design decisions before the domain is completely buildable

1. **Which repo hosts all remaining mining game work (encounters, Explore, everything new)?** Every mining plan pre-dates the rebuild and targets `disbot/`; the rebuild (Q-0241, D-0042–D-0045) has since made superbot-next the successor with mining's deep systems still pending. **Recommended default:** freeze old-bot mining at its current ✔-ready state; execute the D-0043 deep-systems port first (pure execution, rows 2–8); build encounters and all new design on superbot-next only. Building Q-0198 encounters into old-bot `mining_workflow` now would be work ported twice.

2. **Grid navigator (and PIL cards) → next's panel/g1 mapping.** The shipped `MineGridView` (six directional digs, in-place fog-of-war map, PIL cards) has no expression yet in next's PanelSpec/dynamic-session scheme; D-0042 armed the g1 seam but only blackjack/RPS/deathmatch use it. **Recommended default:** grid navigator = a g1 dynamic session keyed `(guild, user, 0, "mining_grid")` per the D-0042 recipe; ship the map as a text/emoji grid first (image `ResultRender` is an undesigned kernel seam — defer it to its own decided slice rather than block the port).

3. **Encounter content table + sim pass (the Q-0198 residue).** Q-0198 decided shape, not content: the per-depth-band event list, loot/hazard magnitudes, and their interaction with energy and the ~1,000–5,000 coins/active-hour band are unauthored. **Recommended default:** v1 = exactly the three Q-0198 archetypes (rich vein / hazard costing energy / coin cache), authored per depth band z 10–20/20+, run through `tools/game_sim/mining_economy_sim.py` extended with an encounter leg before shipping (the Q-0087 sim-first methodology, already precedented twice).

4. **Combat fast-follow engine mapping.** "Reuse the creature/deathmatch engine" is decided but unmapped, and the two candidate engines are in different port states (deathmatch core headless-ported in D-0045 at bare baseline; creature battle engine unported). **Recommended default:** target the ported deathmatch `core.py` (injectable rng, headless — exactly the shape an encounter leg needs) with `EffectiveStats` tilt arriving via the equipment port (row 2); sequence combat *after* rows 2+9 prove out; explicitly rule the creature battle engine out unless the creature-battle port lands first.

5. **Open-world Explore (Roam/Quests) definition — route to the owner.** This is the one genuinely product-shaped hole: the owner himself said the commands "haven't been defined yet," and Quests collides with the AI quest layer (V-10 / Q-0040 bounded-menus) and the AI-cost posture (Q-0190/Q-0082). **Recommended default:** keep the stub; run one dedicated design pass proposing Roam = a thin surface-world flavour loop **sharing the row-11 encounter engine** (design it once), and Quests = deterministic milestone quests first with the AI dungeon-master variant gated behind Q-0048's per-exposure lift; present as structured choices on the question panel.

(Not a design decision but a standing blocker worth flagging: the **paper-doll PNG pack** is an owner *asset* dependency — nothing any agent can design around; and the **mining parity corpus is thin** — 2 goldens for a 37-command surface — so the deep-systems port should mint goldens as it goes.)

**Summary:** Mining is fully shipped and design-closed in the old bot; in superbot-next only the 6-command core loop is live and the 27-command deep-systems mass (D-0043) is pure execution against a pinned oracle — the real invention gaps are just five: repo-of-record for new work, grid/card UI mapping onto next's panel engine, the Q-0198 encounter content table + sim, the combat-engine mapping, and the owner-undefined open-world Explore.
