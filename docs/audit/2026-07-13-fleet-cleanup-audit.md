## What this repo is

`superbot-games` is the shared home of SuperBot's game plugins: mining,
exploration, a D&D-style story game, and fishing, plus the shared world
systems they draw on (inventory, tools, locations, resources, encounters —
`games/shared/`). Per `README.md`, **one seat owns all of `games/**`**
(the historical two-Project mining/exploration split is retired, see
`docs/lanes.md`). Games are built as pure-domain packages meant to ship as
plugins into the rebuilt bot (`menno420/superbot-next`), against a binding
plugin contract (`menno420/superbot-next docs/game-plugin-contract.md@d3dba9b`).
All four games are currently playable stand-alone via
`python3 -m games` / `python3 -m games.mining` / `python3 -m games.fishing`
etc., each driving an "audited workflow seam" (`services/*_workflow.py`)
over an in-memory audit sink (`services/audit.py`).

## CI setup and health

Three workflows in `.github/workflows/`:

- **`tests.yml`** — runs the pure-domain pytest suite, gated by a
  "control fast lane" (a diff touching only `control/**` short-circuits
  green without running tests). The actual test invocation is:
  ```
  python3 tests/check_suite_floors.py
  python3 -m pytest tests/ games/exploration/tests/ -q
  ```
  plus `python3 tools/gen_balance.py --check` (balance-page freshness).
- **`substrate-gate.yml`** — kit-owned hygiene gate (session-card / journal
  checks); regenerated by `bootstrap.py adopt/upgrade`, so hand edits are
  overwritten by design (per its own header comment).
- **`auto-merge-enabler.yml`** — arms native GitHub auto-merge on
  non-draft `claude/*` PRs once required checks are green.

**Finding — CI test-execution gap (verified, not a suspicion):**
`services/tests/` is a **registered** suite in `tests/EXPECTED_SUITES.txt`
and carries its own `EXPECTED_MIN_TESTS.txt` floor (currently **164**), and
`tests/check_suite_floors.py` does *collect* it (`--collect-only`) as part of
the floor guard. But **`tests.yml`'s actual `pytest` invocation never
includes `services/tests/`** — the run line is
`pytest tests/ games/exploration/tests/ -q`, which does not reach
`services/tests`. Verified directly:

```
$ python3 -m pytest tests/ games/exploration/tests/ -q          # what CI actually runs
392 passed

$ python3 -m pytest tests/ games/exploration/tests/ services/tests/ -q   # what several PR bodies claim CI runs
556 passed
```

The 164-test delta is exactly `services/tests`'s floor. Practically: the
floor guard (`check_suite_floors.py`) only *collects* — it never executes —
so it would catch a whole file/dir vanishing, but **a broken assertion
inside an existing `services/tests/test_*.py` file would never turn CI red**,
because that suite is never actually run in `tests.yml`. This matters because
`services/tests/` covers exactly the audited mutation seams
(`dnd_workflow`, `fishing_workflow`, `mining_workflow`, `exploration_workflow`)
that the repo's own binding rules treat as the highest-trust layer. Several
recent PR bodies (#87, #90, #99, #100 among them) quote
`pytest tests/ games/exploration/tests/ services/tests/ -q` as "the suite" —
i.e., authors are already running the wider command locally — but the
committed `tests.yml` was never updated to match. This looks like a one-line
fix (add `services/tests` to the `tests.yml` pytest line) but per the
assigned rules I did not modify CI/workflow files as part of this audit
(out of scope for a docs-only report PR); flagging it here for the next
in-repo session that touches CI.

## Suggestions (2–4, as requested)

1. **Fix the `services/tests` CI-execution gap** (see Finding above) — add
   `services/tests` to `tests.yml`'s pytest invocation line. This is small,
   mechanical, and high-value: it currently means the audited mutation-seam
   tests (dnd/fishing/mining/exploration workflow) are collected-but-not-run
   in the required CI check, so a broken assertion there would not go red.
2. **Centralize the "DARK-repo classification" heuristic across the fleet.**
   This repo shows that a coordinator can go quiet for many hours and then
   produce a tight burst of 5–10+ merges within a single hour, immediately
   before an external sweep runs — a pattern that a simple "no activity in
   N hours as of doc-generation-time" snapshot will systematically
   mis-classify. superbot-games's own `control/outbox.md` NIGHT HEADLINE
   entry (PR #94) shows the in-repo coordinator already independently
   flagged the same stale worklist doc. A fleet-wide convention of
   "always re-check `list_pull_requests`/last-commit-time live, never trust
   a sweep doc's DARK/ACTIVE label without a fresh check" (which this
   audit's own instructions already required) is the right rule — worth
   promoting from "instruction to this one audit" to a documented, reusable
   step in whatever runs the fleet-wide sweep.
3. **Consider a lightweight "is a live session likely running right now"
   signal file** (e.g. a `control/*.lock` or a `last_heartbeat_at` field
   with a short TTL) that other sessions / audits can check cheaply before
   assuming a repo is safe to touch, instead of inferring liveness from
   commit timestamps and control-file prose after the fact. Several repos
   in this fleet (per the fleet context) run cron-driven wake loops; a
   machine-checkable "session likely active" signal would remove the
   guesswork this audit had to do by reading `control/status.md` prose and
   correlating merge timestamps.
4. **Archive-file self-labeling** (see Doc quality above) — a one-line
   "ARCHIVED, see docs/lanes.md" header on the four legacy per-lane
   `control/inbox-*.md` / `control/status-*.md` files would remove the only
   navigation ambiguity found in an otherwise very well-organized `control/`
   directory. Low-risk, mechanical, in-scope for a future session that owns
   `control/**` (not this audit).
